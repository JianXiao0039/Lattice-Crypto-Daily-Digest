Structured fields:

| field | meaning |
| --- | --- |
| `problem_summary` | What problem the paper appears to study, derived from abstract when available. |
| `method_summary` | Method, construction, protocol, attack, implementation, benchmark, or system evidence. |
| `contribution_summary` | Claimed contribution from abstract or conclusion when available. |
| `evidence_basis` | Evidence classes used: `abstract-derived`, `conclusion-derived`, `metadata-derived`, `repository-note-derived`, `title-derived`, or `insufficient-evidence`. |
| `radar_relevance` | Why the paper matters to lattice cryptography, PQC, LWE/RLWE/MLWE, SIS/Module-SIS, BKZ, ML-KEM/ML-DSA, FHE, or AI4LC. |
| `recommendation_reason` | One concise reading action and why. |
| `caveat` | TODO_VERIFY statement for missing abstract, missing conclusion, missing full text, or weak radar evidence. |
| `confidence` | One of `conclusion_supported`, `abstract_supported`, `repository_note_supported`, `metadata_supported`, `title_only`, `insufficient_evidence`. |
| `source_fields_used` | Which record fields were actually used. |
| `todo_verify` | Concrete verification items before treating the rationale as paper-level knowledge. |

Contract:

- The helper does not rank, classify, fetch, schedule, or mutate production artifacts.
- Keyword matches may support relevance but cannot be the whole rationale.
- If conclusion text is absent, rationale must not claim "the conclusion says".
- If abstract text is absent, method and contribution stay low-confidence.
- Weak FHE application papers should be marked peripheral/temporary when the evidence is mainly application or system context.
