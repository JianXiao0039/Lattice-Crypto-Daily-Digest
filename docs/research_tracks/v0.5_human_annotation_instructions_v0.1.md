# v0.5 Human Annotation Instructions v0.1

## Purpose

Review the compact 25-record annotation pack without changing production classification. The Markdown file is convenient for reading; CSV or JSON may be used for recording decisions.

## Allowed Decisions

- `user_confirmed`: the reviewed primary and secondary tracks are correct.
- `user_corrected`: the user supplied corrected tracks.
- `excluded_insufficient_metadata`: repository evidence cannot support a label.
- `conflict_requires_adjudication`: evidence supports competing interpretations.

Only the user may assign `user_confirmed` or `user_corrected`.

## Review Procedure

1. Read the exact title and retained abstract/evidence.
2. Compare production, shadow, and Codex-reviewed labels independently.
3. Require a technical lattice/PQC anchor; title keywords alone are insufficient.
4. Fill the human-gold primary/secondary fields, reviewer note, and status.
5. Use `TODO_VERIFY` where the retained metadata cannot establish the claim.

## Track Guardrails

- Track A rejects generic signatures and commitments without SIS/Module-SIS/lattice evidence.
- Track B is based on technical relevance, never author identity alone.
- Track C rejects generic ML without a lattice-cryptanalysis interface.
- Track D requires concrete PQC/FHE/implementation evidence and is not a miscellaneous bucket.

No true precision/recall metric is valid until user-confirmed or user-corrected labels exist.
